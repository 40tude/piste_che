# Implementation Plan: Resort Data Generator CLI

**Branch**: `002-resort-generator` | **Date**: 2026-03-19 | **Spec**: specs/002-resort-generator/spec.md
**Input**: Feature specification from `/specs/002-resort-generator/spec.md`

## Summary

Single CLI binary (`resort_generator`) added as a Cargo workspace member. Accepts `--resort "<Name>"`, fetches trail geometry from Overpass API + elevation from IGN Altimetrie API, merges results, writes timestamped JSON to `data/`. Ports and unifies two existing prototypes (`get_data` + `get_elevation` from `020_serre_che_1`).

## Technical Context

**Language/Version**: Rust (edition 2024, stable)
**Primary Dependencies**: clap 4 (derive), reqwest 0.13 (json), serde/serde_json 1, chrono 0.4, anyhow 1, tokio 1, tracing 0.1, tracing-subscriber 0.3
**Storage**: JSON files in `data/` folder
**Testing**: cargo test; mockall for HTTP mocking
**Target Platform**: Windows 11 local workstation
**Project Type**: CLI binary (Cargo workspace member)
**Performance Goals**: Complete run < 5 minutes per resort
**Constraints**: IGN batch size 50, 200 ms inter-batch delay, 3 retries with exponential backoff
**Scale/Scope**: Single operator, infrequent manual runs

Root `Cargo.toml` updated to declare workspace:

```toml
[workspace]
members = ["resort_generator"]
```

**Structure Decision**: Flat single-crate CLI with four modules. No shared type crate -- the JSON file format is the contract between generator and main app. Types are defined independently in each crate (generator writes Overpass-compatible JSON; main app reads it via `OsmData::load`). This avoids coupling the lightweight CLI to the Leptos/WASM dependency tree.
